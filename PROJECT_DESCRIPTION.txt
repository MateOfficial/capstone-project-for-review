1. ЧТО ЭТО ЗА ПРОЕКТ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

StoreFlow — это полноценное веб-приложение для управления музыкальным магазином.
Проект разработан под реальный бизнес (магазин Yamaha) и покрывает весь цикл
работы магазина:

  • Публичный каталог товаров для покупателей (с AI-консультантом)
  • Административная панель для сотрудников и менеджеров
  • Управление складом, ценами, категориями товаров
  • CRM — база клиентов
  • HR — сотрудники, расписания, посещаемость
  • Документооборот — гарантии, акты выдачи, PDF-генерация
  • Финансовые отчёты
  • AI-помощник на основе локальной языковой модели

Проект спроектирован как SaaS — то есть одна система может обслуживать
несколько магазинов одновременно (мульти-тенантность через store_id).

  BACKEND:
    • Java 21
    • Spring Boot 3 (веб-фреймворк)
    • Spring Security (авторизация и права доступа)
    • Spring Data JPA / Hibernate (работа с базой данных)
    • PostgreSQL 16 (основная база данных)
    • Redis 7 (кэш и хранение сессий)
    • JWT (JSON Web Tokens) — аутентификация без сессий
    • iText / PDF (генерация PDF-документов)
    • Maven (сборка проекта)

  FRONTEND:
    • React 18 (UI-фреймворк)
    • TypeScript (типизированный JavaScript)
    • Vite (сборщик, dev-сервер)
    • Nginx (веб-сервер в production для раздачи файлов)

  AI / МАШИННОЕ ОБУЧЕНИЕ:
    • Ollama (локальный сервер для запуска языковых моделей)
    • qwen2.5:3b (языковая модель — работает на русском языке)

  ИНФРАСТРУКТУРА:
    • Docker (контейнеризация каждого сервиса)
    • Docker Compose (оркестрация всех контейнеров)

  ПРИНЦИП РАБОТЫ КАЖДОГО ЗАПРОСА:
  ─────────────────────────────────
  1. HTTP запрос приходит на Controller
  2. Controller проверяет авторизацию (@PreAuthorize)
  3. Controller вызывает Service с бизнес-логикой
  4. Service использует Repository для работы с БД
  5. Repository — это интерфейс JPA (Spring Data), SQL генерируется автоматически
  6. Ответ упаковывается в ApiResponse{success, data, error, timestamp}
  7. Возвращается клиенту в JSON формате

6. КАК РАБОТАЕТ АВТОРИЗАЦИЯ (JWT)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

JWT (JSON Web Token) — это подход без хранения сессий на сервере.

  Процесс входа:
  ──────────────
  1. Пользователь вводит логин/пароль
  2. Backend проверяет пароль (bcrypt хеш в БД)
  3. Если верно — возвращает два токена:
     • Access Token (живёт 15 минут) — для обычных запросов
     • Refresh Token (живёт 30 дней) — для обновления access token
  4. Frontend сохраняет токены в localStorage
  5. Каждый запрос к /api/admin/** отправляется с заголовком:
     Authorization: Bearer <access_token>
  6. Когда access token истекает — frontend автоматически обновляет
     его через /api/auth/refresh с refresh token

  Права доступа:
  ──────────────
  • Каждый пользователь имеет Роль (например: ADMIN, MANAGER, EMPLOYEE)
  • Роль содержит набор Разрешений (Permission):
    - catalog.manage — управление товарами
    - hr.manage — управление сотрудниками
    - documents.manage — работа с документами
    - reports.view — просмотр отчётов
    - users.manage — управление пользователями
  • Методы помечаются @PreAuthorize("hasAuthority('catalog.manage')")
  • Если права нет — возвращается 403 Forbidden

7. ОТЧЁТЫ И ИНТЕГРАЦИИ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Система поддерживает отчёты и интеграции без AI-компонентов.

  Компоненты:
  ───────────
  • ReportsPage.tsx — отчёты и выгрузки CSV
  • ReportController — агрегированные показатели и выборки
  • IntegrationMappingPage.tsx — настройка маппинга полей для импорта
  • IntegrationImportService — загрузка и обработка данных из CSV/Excel

  Пошаговый процесс отчётов:
  ──────────────────────────
  Шаг 1. Администратор выбирает период (from/to)
  Шаг 2. Backend формирует агрегаты по продажам/сотрудникам/посещаемости
  Шаг 3. UI отображает карточки показателей и таблицы
  Шаг 4. При необходимости данные выгружаются в CSV
         "Каталог магазина (показано 23 из 1396 товаров):
          - PSSE30 [KL Синт, детский] | 1170000 сум | есть (5 шт)
          - PSSF30 [KL Синт, детский] | 1170000 сум | нет в наличии
          - YPT270 [KL Синт, подростковый] | 2830000 сум | есть (3 шт)
          ..."

  Шаг 7. Собираем полный запрос к Ollama:
         System prompt: "Ты консультант магазина Yamaha. Каталог товаров
           передан в формате: Название [Категория] | Цена | Наличие.
           Коды товаров — модели производителей (Yamaha, Shure и др.)
           Используй свои знания о брендах. Отвечай только по каталогу..."
         User message: "Запрос клиента: посоветуй синтезатор... + контекст"

  Шаг 8. Ollama генерирует ответ (на CPU занимает ~2 мин)
         → "PSSE30 — детский синтезатор Yamaha, 1 170 000 сум, есть в наличии.
            Отличный выбор для начинающих: 32 клавиши, встроенные уроки..."

  Шаг 9. Ответ возвращается пользователю в чат

  Почему локальный LLM (не ChatGPT):
  ────────────────────────────────────
  • Данные каталога не уходят в интернет (конфиденциальность)
  • Нет затрат на API (ChatGPT ~$0.01 за запрос)
  • Работает без интернета
  • Полный контроль над моделью

8. БАЗА ДАННЫХ — ОСНОВНЫЕ ТАБЛИЦЫ

  warranty_rules      — правила гарантии по категориям
  issuance_acts       — акты выдачи товара
  document_templates  — шаблоны документов

  settings            — ключ-значение настройки магазина
  module_configs      — вкл/выкл модулей (CRM, HR, Documents...)
  audit_logs          — журнал всех действий в системе
  refresh_tokens      — активные refresh токены пользователей

  Схема создаётся автоматически через Hibernate при первом запуске.
  Начальные данные (роли, права, тестовый администратор) вставляются
  через DataInitializer при старте приложения.

10. КЛЮЧЕВЫЕ ТЕХНИЧЕСКИЕ РЕШЕНИЯ И ПАТТЕРНЫ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  1. МУЛЬТИ-ТЕНАНТНОСТЬ
     Каждый объект в БД (товар, клиент, сотрудник) содержит store_id.
     StoreResolver автоматически определяет магазин из JWT токена или
     из публичных настроек. Один сервер — много магазинов.

  2. ЕДИНЫЙ ФОРМАТ API ОТВЕТОВ
     Все эндпоинты возвращают одинаковую структуру:
     {
       "success": true/false,
       "data": { ... },           // при успехе
       "error": { "code": "...", "message": "..." }, // при ошибке
       "timestamp": "2026-05-07T..."
     }
     GlobalExceptionHandler перехватывает все исключения и упаковывает
     их в этот формат автоматически.

  3. ЛЕНИВАЯ ЗАГРУЗКА JPA + @Transactional
     Связанные объекты (например product.category) загружаются
     только при обращении — это называется LAZY loading.
     Чтобы это работало внутри HTTP запроса, методы помечаются
     @Transactional(readOnly = true) — Hibernate сессия остаётся
     открытой на время всего запроса.
     Для сложных запросов используется LEFT JOIN FETCH в JPQL чтобы
     загрузить связанные данные одним SQL запросом (без N+1 проблемы).

  4. REST API ДИЗАЙН
     Публичные эндпоинты:  /api/public/**  (без авторизации)
     Пользовательские:     /api/**         (нужен JWT)
     Административные:     /api/admin/**   (нужны права)

  5. КОНТЕЙНЕРИЗАЦИЯ
     Каждый сервис — отдельный Dockerfile:
     backend/Dockerfile  — многоэтапная сборка (Maven build → JRE image)
     frontend/Dockerfile — сборка Vite → копирование в Nginx image
     Итоговые образы минимального размера, без лишних инструментов.

  6. УМНАЯ ФИЛЬТРАЦИЯ ДЛЯ AI
     LLM имеет ограниченное "окно памяти" (context window = 8192 токена).
     1396 товаров полностью не вмещаются.
     Решение: перед отправкой в LLM фильтруем каталог по ключевым словам
     из вопроса пользователя. В контекст попадают только релевантные товары.
     Fallback: если по ключевым словам найдено менее 5 товаров — берём
     первые 80 из каталога.

  7. FUZZY MATCHING ПРИ ИМПОРТЕ
     При загрузке товаров из внешних источников (CSV, Excel) названия
     могут отличаться ("Yamaha PSR-E383" vs "PSRE383").
     FuzzyMatcher использует алгоритм расстояния Левенштейна для поиска
     наиболее похожего товара в существующем каталоге.

  8. АУДИТ ВСЕХ ДЕЙСТВИЙ
     AuditService автоматически логирует все изменения (создание, редактирование,
     удаление) с указанием пользователя, времени и IP адреса.
     Администратор может просмотреть полный журнал в AuditPage.
